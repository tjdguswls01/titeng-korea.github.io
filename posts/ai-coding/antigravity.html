01 · Definition

IDE를 넘어, 에이전트 작업 공간으로

발표 범위: IDE
  • Google이 제공하는 에이전트 중심 개발 플랫폼의 IDE 제품
  • 코드 편집기, Agent Manager, Artifacts, 코드베이스 이해를 한 환경에 통합
  • 에이전트가 편집기·터미널·브라우저를 사용해 계획부터 검증까지 수행
  • 플랫폼에는 CLI와 SDK도 있지만, 이번 발표는 IDE 경험에 집중

“어떤 코드를 쓸까?”보다 “어떤 결과와 증거를 요구할까?”가 중요해진다.

02 · Core difference

Agent Manager: 여러 작업을 관제한다

① Parallel work
  • 여러 작업 공간과 에이전트의 상태를 한 화면에서 관리
  • 독립적인 작업은 나눠 맡기고 비동기로 진행
  • 필요한 순간 Editor View로 전환해 직접 개입
  • 의존성이 낮은 작업에서 병렬화 효과가 크다
목표는 에이전트 수가 아니라, 좋은 작업 분해와 명확한 검토 지점이다.

06 · First task

좋은 요청 = 목표 + 범위 + 제약 + 완료 기준 + 검증

Prompt as specification
모호한 요청

작업의 경계와 성공 조건이 없다

“로그인 기능 만들어줘”

에이전트가 범위·API·검증 방식을 추측하게 된다.

구조화된 요청

완료를 판단할 수 있는 기준을 준다

`src/auth` 범위에서 이메일 로그인 오류를 수정하라. 공개 API는 유지하고, 실패 테스트를 먼저 재현한 뒤 관련 테스트와 린트를 통과시켜라. 변경 파일과 검증 결과를 Artifact로 요약하라.
목표범위제약완료 기준검증

실무 원칙: 큰 작업은 계획과 구현을 분리하고, 먼저 Plan Artifact를 검토한 뒤 실행합니다.

에이전트의 품질은 모델뿐 아니라 문제 정의와 완료 기준의 품질에 크게 좌우된다. 먼저 계획만 요청하는 습관을 권한다.

삭제, 배포, 자격 증명 접근, 외부 전송 같은 고위험 작업에는 명시적인 승인 단계를 둔다. 커밋 전 최종 책임은 사람에게 있다.

08 · Safe parallelism

병렬화의 기준은 독립성이다

Speed ≠ no coordination
병렬화 적합

서로의 결과를 기다리지 않는 작업

  • 서로 다른 모듈 조사
  • 독립적인 테스트 작성
  • 문서 초안과 대안 비교
  • 각기 다른 작업 공간의 변경
순차 실행 필요

선행 결과가 다음 작업의 입력인 경우

  • 공통 인터페이스 변경 → 소비 코드 수정
  • DB 마이그레이션 → 배포
  • 같은 파일을 동시에 수정
  • 공통 설정을 바꾸는 작업

실무 원칙: 속도를 높이기 전에 충돌 비용과 리뷰 부담을 계산합니다.

각 에이전트에 파일 범위, 산출물, 금지 사항, 완료 기준을 명확히 지정한다. 작은 범위와 별도 브랜치 또는 작업 공간을 권한다.

11 · Decision

도구가 아니라 업무 적합성을 선택한다

Evaluate with real work

Antigravity가 잘 맞는 경우

  • 병렬 작업과 Artifact 리뷰를 기본 흐름으로 사용
  • 웹·풀스택의 브라우저 검증을 자주 반복
  • Google·Gemini 중심 환경과의 결합이 중요

VS Code가 잘 맞는 경우

  • 기존 확장·디버거·Dev Container·조직 표준 유지
  • 모델·에이전트·확장을 세밀하게 조합
  • 학습 비용과 운영 안정성이 더 중요
가장 좋은 평가법: 작은 실제 과제를 동일한 완료 기준으로 수행하고 품질 · 시간 · 검토 부담을 함께 측정한다.
발표 시점 공식 페이지 확인 필요: 버전, 요금, 지원 모델, 사용량 제한은 변경될 수 있습니다.
