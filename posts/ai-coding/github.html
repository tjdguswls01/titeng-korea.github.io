Git × VS Code × AI Agent

Git 명령어에서
프롬프트로

명령어를 없애는 것이 아니라, 작업 의도와 안전 조건을 더 정확히 전달하는 방식

18 slides · practical guide
$ git status
$ git diff --staged
$ git commit -m "fix: 로그인 오류 수정"
PROMPT현재 변경을 확인하고 로그인 오류 수정과 관련된 파일만 stage해줘. staged diff와 테스트 결과를 요약한 뒤, 커밋 전에는 내 승인을 기다려줘.

참고: git-commands-slides.html · Git 명령어 20가지: 명령어에서 프롬프트로

명령어가 사라진다는 주장이 아니다. 작업 의도와 안전 조건을 프롬프트로 전달하고, 사람이 결과를 검토하는 변화다.

과거에는 명령어를 골랐고,

Interface shift

명령어 중심

필요한 명령어와 옵션을 기억해 직접 실행했습니다.

status → diff → add → commit → push
→
현재 · 프롬프트 중심

의도와 안전 조건 중심

원하는 결과, 변경 범위, 멈춤 조건, 검증 방법을 설명하고 승인합니다.

목표 → 범위 → 안전 조건 → 검증 → 승인

변화: 명령어 암기 부담 감소 · 작업 의도와 검토 책임 증가

예전에는 정확한 옵션을 아는 것이 첫 단계였다. 지금은 무엇을 왜 바꾸는지, 언제 멈출지 설명하는 것이 첫 단계다.

VS Code 안에서 변경·대화·검토가 연결된다

Source Control + Agent

  • 터미널 명령과 diff를 같은 문맥에서 검토

한 화면에 모였다고 검토가 끝나는 것은 아니다. 파일 상태와 staged diff를 직접 확인한다.

03 · New workflow

명령어 나열보다 검토 가능한 흐름을 요청한다

Understand → delegate → verify
01상태 확인branch·status
02변경 검토diff·위험 파일
03범위 선택stage 대상
04검증test·lint
05승인commit·push
WORKFLOW PROMPT현재 상태를 먼저 확인해줘. 변경을 기능별로 분류하고 위험한 파일을 표시해줘. 테스트 후 커밋 계획을 제안하되, stage와 commit 전에는 승인을 요청해줘.

Git 원리를 이해하고, 읽기 작업 → 계획 → 검증 → 승인 → 변경 작업 순서를 지킵니다.

프롬프트는 Git을 대체하지 않는다. Git 상태를 이해한 뒤 읽기 전용 진단과 변경 작업 사이에 승인 지점을 둔다.

04 · Repository state

git status — 상태를 보는 명령에서
위험을 찾는 질문으로

Read only
기존 사용
git status
git status -sb
수정·추가·삭제·미추적 파일을 직접 읽었습니다.
현재 프롬프트
PROMPT현재 저장소 상태를 표로 요약해줘. 비밀키, 빌드 산출물, 대용량 파일처럼 커밋하면 안 될 가능성이 있는 항목은 따로 표시해줘.
확인: 현재 브랜치, untracked 파일, 삭제 파일

참고 명령: git status · git branch --show-current

상태 확인은 가장 안전한 시작점이다. 변경하지 말고 요약만 하라고 명시한다.

05 · Review changes

git diff — AI 설명보다
변경 자체를 읽는다

Evidence first
기존 사용
git diff
git diff --staged
git diff src/auth.js
파일별 diff를 직접 읽었습니다.
현재 프롬프트
PROMPT변경을 의도된 변경, 위험한 변경, 확인 질문으로 나눠 요약해줘. 테스트 삭제, 설정 변경, 공개 API 변경은 강조해줘.
확인: 삭제·설정·테스트 diff는 직접 다시 봅니다.

검토 기준: scope · behavior · tests · configuration · secrets

완료 보고보다 diff가 핵심 증거다.

06 · Select scope

git add — 파일 추가에서
커밋 경계 설계로

Stage intentionally
기존 사용
git add src/auth.js
git add -p
# 주의: git add .
파일 또는 hunk를 직접 선택했습니다.
현재 프롬프트
PROMPT이번 목적은 로그인 오류 문구 수정이야. 관련 변경만 stage하고 제외 파일과 이유를 알려줘. 새 파일과 삭제 파일은 먼저 확인해줘.
확인: git diff --staged

이번 목적과 관련된 변경만 stage합니다.

git add .은 불필요한 파일까지 포함할 수 있다.

07 · Record intent

git commit — 저장이 아니라
설명 가능한 기록

History as documentation
기존 사용
git commit -m "fix login"
git commit
직접 메시지를 작성하고 커밋했습니다.
현재 프롬프트
PROMPTstaged diff 기반 한국어 Conventional Commit 메시지 3개를 추천해줘. 내가 하나를 선택한 뒤 커밋해줘.
확인: staged diff와 메시지가 같은 의도인가?

무엇을 바꿨고, 왜 바꿨으며, 무엇은 바꾸지 않았는가?

AI 메시지도 실제 변경과 대조한다.

08 · Isolate work

branch · switch · worktree
작업 전에 격리한다

Protect main
git branch --show-current
git switch -c feature/login
git worktree add ../login-work feature/login
PROMPT현재 상태와 기준 브랜치를 확인해줘. 기존 변경과 충돌할 수 있으면 새 브랜치 또는 worktree로 격리해줘. 미커밋 변경이 있으면 이동하지 말고 알려줘.
병렬 에이전트는 같은 파일을 동시에 수정하지 않게 격리합니다.

격리 기준: 독립 작업 · 장기 작업 · 실험 · 병렬 에이전트

에이전트를 병렬로 사용할수록 worktree가 중요하다.

09 · Sync safely

fetch · pull — 원격 변경을
확인한 뒤 가져온다

Inspect first

1. fetch

현재 작업을 바꾸지 않고 원격 상태만 가져온다.

git fetch --prune

2. compare

ahead/behind와 들어올 커밋을 확인한다.

git log HEAD..origin/main

3. integrate

전략을 확인한 뒤 pull한다.

git pull --ff-only
PROMPT원격 상태를 fetch하고 현재 브랜치와 차이를 요약해줘. 충돌 가능성이 있거나 fast-forward가 불가능하면 pull하지 말고 선택지를 설명해줘.

fetch → compare → choose strategy → pull

pull 전에 원격 변경과 통합 전략을 확인한다.

10 · Conflict

충돌은 자동 제거 대상이 아니라
의도 차이를 확인하는 대화다

Do not guess
  • 충돌 파일과 양쪽 의도를 먼저 요약
  • 설정·API 충돌은 자동 선택 금지
  • 버려지는 변경을 명확히 표시
  • 해결 후 테스트와 diff 재확인
PROMPTours와 theirs의 의도를 설명해줘. 임의 선택하지 말고 해결안과 각 해결안에서 사라지는 변경을 보여줘. 내가 방향을 선택한 뒤 수정해줘.
금지: 충돌 마커를 기계적으로 삭제하거나 설정을 추측으로 합치기

understand → choose → edit → test → review

문법적 해결과 비즈니스 의도 해결은 다르다.

11 · Undo safely

restore · reset · revert
무엇을 보존할지 먼저 묻는다

High risk
도구목적위험프롬프트 조건
restore작업 트리·stage 복원미커밋 작업 손실손실 내용을 먼저 표시
resetstage·커밋 위치 이동옵션별 파일·히스토리 손실soft/mixed/hard 비교 후 승인
revert공유 커밋을 새 커밋으로 취소충돌·부분 되돌림대상과 영향 범위 확인
PROMPTrestore, reset, revert를 비교해줘. 작업 손실과 원격 공유 여부를 확인하고, 가장 안전한 방법을 추천하되 실행 전 정확한 결과를 보여줘.

공유 히스토리는 일반적으로 reset보다 revert가 안전합니다.

reset --hard는 승인 없이 실행하지 않는다.

12 · Share changes

push · pull request
보내기 전에 리뷰 문맥을 만든다

External effect

Push 전 확인

  • 현재 브랜치와 upstream
  • 새로 올라갈 커밋
  • 테스트·린트·빌드
  • force push 여부
PROMPT원격에 올라갈 커밋과 변경 범위를 요약해줘. 테스트가 통과하면 PR 제목과 본문을 만들고, push와 PR 생성은 승인 후 실행해줘.
PR: 목적 · 주요 변경 · 검증 · 리뷰 포인트 · 위험

push · PR 생성 · force push는 명시적 승인 후 실행

PR 설명도 실제 diff와 대조한다.

13 · Guardrails

좋은 Git 프롬프트에는
멈춤 조건이 들어간다

Stop before damage

작업 손실

미커밋 변경을 버릴 가능성

히스토리 재작성

reset·rebase·force push

범위 초과

요청 밖 파일·설정 변경

외부 영향

push·PR·tag·release

공통 문장: 작업 손실, 히스토리 재작성, 원격 변경, 범위 밖 수정 가능성이 있으면 실행하지 말고 먼저 보고해줘.

“무엇을 해줘”만큼 “언제 멈춰줘”가 중요합니다.

기본 승인 정책 외에 프롬프트에도 중단 조건을 넣는다.

14 · Practical example

같은 작업, 다른 인터페이스

Login bug fix
기존 사용

명령어를 직접 실행

git switch -c fix/login
git status
git diff
git add src/auth.js
git commit -m "fix login"
git push -u origin fix/login
→
현재 사용

의도와 안전 조건을 프롬프트로

PROMPT로그인 오류 수정을 별도 브랜치에서 진행해줘. src/auth 범위만 수정하고 테스트해줘. 관련 변경만 stage한 뒤 계획을 보여주고, commit과 push 전에는 승인받아줘.

명령어 수보다 의도·범위·검증·승인의 문맥이 중요합니다.

에이전트가 어떤 명령을 실행했는지 확인할 수 있어야 한다.
