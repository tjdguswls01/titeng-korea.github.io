# AI 시대의 Git 활용법 with VSCode 발표 대본

## 발표 기준

- 예상 분량: 약 40분
- 기준 자료: `topic2.html`
- 진행 방식: 슬라이드당 평균 1분 내외, 핵심 슬라이드는 1분 30초까지 설명
- 발표 톤: Git 명령어 강의가 아니라, AI가 만든 변경을 안전하게 검토하는 실무 습관 중심

## 슬라이드별 대본

### 1. 제목

안녕하세요. 이번 발표에서는 “AI 시대의 Git 활용법 with VSCode”를 다룹니다.  
요즘 AI는 코드를 굉장히 빠르게 만들어 줍니다. 하지만 코드가 빨리 만들어진다는 것은, 그만큼 우리가 더 빠르게 변경을 이해하고 검토하고 설명해야 한다는 뜻이기도 합니다.

오늘의 핵심은 Git을 어렵게 배우자는 것이 아닙니다. AI가 만든 결과를 무조건 믿지 않고, Git과 VSCode를 이용해서 변경을 확인하고, 필요한 것만 고르고, 나중에 되돌릴 수 있는 형태로 남기는 습관을 만드는 것입니다.

### 2. 오늘의 질문

먼저 질문 하나로 시작하겠습니다. AI가 코드를 더 빨리 만들면, 우리는 그 코드를 어떻게 믿을 수 있을까요?  
정답은 “AI니까 믿는다”가 아닙니다. 반대로 “AI는 못 믿으니까 쓰지 않는다”도 아닙니다.

현실적인 답은 Git을 기준으로 변경을 통제하는 것입니다. Git은 AI가 만든 결과를 사람이 검토할 수 있는 단위로 쪼개고, 어떤 변경이 들어왔는지 보여주고, 문제가 생겼을 때 되돌릴 수 있게 해줍니다.

### 3. 발표 목표

오늘 발표의 목표는 세 가지입니다.  
첫째, Git이 단순한 백업 도구가 아니라 변경을 설명하는 도구라는 점을 이해합니다.  
둘째, VSCode Source Control 패널에서 파일 상태, diff, stage, commit을 읽는 방법을 익힙니다.  
셋째, AI 코딩 결과를 검토하고 PR로 전달하는 실무 루틴을 정리합니다.

이 발표가 끝나면 “AI가 고쳐 줬으니 끝”이 아니라, “무엇이 바뀌었고, 왜 바뀌었고, 어떻게 검증했는지”를 설명할 수 있어야 합니다.

### 4. 핵심 결론 미리보기

오늘의 핵심 결론을 먼저 말씀드리면, AI 시대의 개발자는 코드를 직접 많이 치는 사람에서 변경을 잘 관리하는 사람으로 역할이 이동합니다.  
AI는 초안을 빠르게 만들 수 있습니다. 하지만 그 초안이 제품 코드에 들어가도 되는지 판단하는 것은 여전히 개발자의 일입니다.

그래서 Git은 더 중요해집니다. Git은 변경을 작게 나누고, 비교하고, 저장하고, 공유하고, 되돌리는 기준점이기 때문입니다.

### 5. Git의 역할 1: Git은 백업 도구가 아니다

Git을 처음 배울 때 흔히 “파일을 저장하는 도구”처럼 이해합니다. 물론 저장 기능도 있습니다. 하지만 실무에서 Git의 진짜 가치는 변경의 이유와 맥락을 기록하는 데 있습니다.

AI가 만든 코드는 겉으로 보면 동작할 수 있습니다. 하지만 “왜 이 줄이 추가됐는지”, “기존 로직을 왜 삭제했는지”, “테스트는 어떤 기준으로 통과했는지”가 남지 않으면 팀에서 유지보수하기 어렵습니다.

Git은 코드의 시간표이자 설명서입니다.

### 6. Git이 관리하는 세 가지

Git이 관리하는 핵심은 세 가지입니다.  
첫째, 파일의 현재 상태입니다. 어떤 파일이 수정되었는지, 새로 생겼는지, 삭제되었는지를 보여줍니다.  
둘째, 변경의 차이입니다. 이전 상태와 현재 상태를 줄 단위로 비교해 줍니다.  
셋째, 저장된 이력입니다. 커밋이라는 단위로 변경을 설명 가능한 기록으로 남깁니다.

이 세 가지를 이해하면 VSCode의 Source Control 화면이 단순한 파일 목록이 아니라, 현재 프로젝트의 변경 대시보드처럼 보이기 시작합니다.

### 7. 코드 작성 속도보다 변경 검토 속도

AI는 30초 만에 여러 파일을 고칠 수 있습니다. 그런데 우리가 그 변경을 30초 만에 검토할 수 없다면, 생산성은 실제로 올라가지 않습니다.

AI 시대의 병목은 타이핑 속도가 아니라 검토 속도입니다. 무엇이 바뀌었는지, 그 변경이 요구사항과 맞는지, 부작용은 없는지 확인하는 능력이 더 중요해집니다.

Git diff를 읽는 습관은 이 병목을 줄이는 가장 기본적인 기술입니다.

### 8. 겉으로 멀쩡해 보이는 코드의 위험

AI가 만든 코드는 문법적으로는 깔끔해 보일 수 있습니다. 변수명도 그럴듯하고, 함수도 잘 나뉘어 있을 수 있습니다. 하지만 위험은 그 안에 숨어 있습니다.

예를 들어 기존 예외 처리를 삭제했을 수도 있고, 특정 케이스에서만 필요한 조건문을 단순화했을 수도 있습니다. 테스트가 없는 영역에서는 이런 변경이 바로 드러나지 않습니다.

따라서 AI 결과를 볼 때는 “코드가 예쁜가”보다 “기존 동작을 깨지 않았는가”를 먼저 봐야 합니다.

### 9. 오늘의 네 가지 원칙

오늘 다룰 원칙은 네 가지입니다.  
첫째, 작업 전 상태를 깨끗하게 만든다.  
둘째, diff를 먼저 읽는다.  
셋째, stage로 커밋 범위를 고른다.  
넷째, commit과 PR에 변경 이유를 남긴다.

이 네 가지만 지켜도 AI가 만든 코드가 섞이면서 발생하는 많은 문제를 줄일 수 있습니다.

### 10. VSCode Source Control 패널

VSCode의 Source Control 패널은 매일 보는 Git 대시보드입니다. 여기에는 변경된 파일 목록, 변경 상태, stage 여부가 표시됩니다.

중요한 것은 이 패널을 “커밋 버튼 누르는 곳”으로만 보지 않는 것입니다. Source Control은 현재 작업 공간이 깨끗한지, AI가 어떤 파일을 건드렸는지, 예상하지 못한 파일이 생겼는지 확인하는 첫 번째 화면입니다.

AI 작업 전과 후에는 반드시 이 패널을 확인하는 습관을 들이는 것이 좋습니다.

### 11. 파일 상태 읽기

Source Control에 보이는 M, U, A, D 같은 글자는 변경 상태를 알려주는 표지판입니다.  
M은 Modified, 이미 Git이 추적하던 파일이 수정되었다는 뜻입니다.  
U는 Untracked, 새 파일이 생겼지만 아직 Git이 추적하지 않는다는 뜻입니다.  
A는 Added, 새 파일이 stage되어 커밋 후보가 되었다는 뜻입니다.  
D는 Deleted, 추적 중이던 파일이 삭제되었다는 뜻입니다.

AI가 작업한 뒤에는 특히 U와 D를 주의해서 봐야 합니다. 새 파일이 불필요하게 생기거나, 중요한 파일이 삭제되었을 수 있기 때문입니다.

### 12. Diff 보기

AI 결과를 믿기 전에 변경 자체를 읽어야 합니다. 이때 보는 화면이 diff입니다.  
diff는 단순히 “추가된 줄”만 보는 것이 아닙니다. 삭제된 줄, 수정된 조건, 바뀐 import, 변경된 설정값까지 모두 봐야 합니다.

좋은 검토는 “AI가 무엇을 추가했는가”가 아니라 “기존 코드에서 무엇을 바꾸었는가”를 보는 데서 시작합니다.

### 13. Diff에서 확인할 것

diff를 볼 때는 몇 가지 질문을 던지면 좋습니다.  
삭제된 코드가 정말 필요 없어진 코드인가?  
조건문이 단순해졌다면 예외 케이스가 사라지지 않았는가?  
테스트 코드가 실제 요구사항을 검증하고 있는가?  
이름 변경이나 포맷 변경이 기능 변경과 섞여 있지 않은가?

이 질문들은 코드 리뷰에서도 그대로 사용할 수 있습니다.

### 14. Stage

Stage는 이번 커밋에 담을 변경을 고르는 장바구니입니다.  
작업 디렉터리에 변경이 여러 개 있어도, 이번 커밋에 모두 담을 필요는 없습니다.

AI가 한 번에 여러 파일을 고쳤다면 더 중요합니다. 기능 수정, 포맷 변경, 문서 수정이 섞여 있으면 리뷰하기 어렵습니다. stage를 이용해 의미 있는 단위로 나누면 커밋 히스토리가 훨씬 읽기 좋아집니다.

### 15. Partial staging

한 파일 안에서도 일부 변경만 커밋할 수 있습니다. 이것을 partial staging이라고 생각하면 됩니다.  
예를 들어 한 파일에서 버그 수정과 포맷 변경이 함께 발생했다면, 버그 수정 부분만 먼저 stage할 수 있습니다.

VSCode diff 화면에서는 변경 블록 단위로 stage할 수 있습니다. 이 기능을 사용하면 “한 파일은 하나의 커밋”이라는 제한에서 벗어나, “하나의 의도는 하나의 커밋”이라는 기준을 만들 수 있습니다.

### 16. Commit

Commit은 저장이 아니라 설명입니다.  
커밋 메시지는 나중에 누군가가 이 변경을 볼 때 가장 먼저 읽는 문장입니다. “update”나 “fix”만 적힌 커밋은 거의 도움이 되지 않습니다.

좋은 커밋은 무엇을 바꿨는지뿐 아니라 왜 바꿨는지를 알려줍니다. AI가 만든 코드도 마찬가지입니다. 누가 코드를 썼는지가 아니라, 어떤 의도로 저장했는지가 중요합니다.

### 17. 전환점

여기까지가 Git 기본기의 첫 번째 묶음입니다. 이제부터는 이 기본기를 AI 작업 흐름에 어떻게 연결할지 보겠습니다.

AI를 잘 쓰는 방법은 더 긴 프롬프트를 쓰는 것이 전부가 아닙니다. Git 상태를 깨끗하게 만들고, 작은 단위로 요청하고, diff를 읽고, 커밋과 PR로 설명하는 전체 루틴이 필요합니다.

### 18. AI 작업 전 체크포인트

AI에게 작업을 맡기기 전에 먼저 `git status`를 확인합니다.  
이미 내 작업이 남아 있는데 AI에게 수정을 맡기면, 내가 한 변경과 AI가 한 변경이 섞입니다. 그러면 나중에 어떤 변경이 누구의 의도였는지 구분하기 어렵습니다.

그래서 작업 전에는 커밋하거나, stash하거나, 별도 브랜치로 옮긴 뒤 시작하는 것이 안전합니다.

### 19. 좋은 흐름

좋은 흐름은 단순합니다. 먼저 현재 상태를 확인합니다. 그다음 AI에게 작은 작업을 맡깁니다. 결과가 나오면 diff를 읽습니다. 테스트를 실행합니다. 문제가 없으면 stage하고 commit합니다.

이 흐름의 장점은 중간에 문제가 생겨도 어디서부터 잘못됐는지 찾기 쉽다는 것입니다. AI가 많은 일을 한 번에 처리할수록, 우리는 더 작은 검토 단위가 필요합니다.

### 20. 요청 범위

AI에게 한 번에 너무 많이 맡기지 않는 것이 중요합니다.  
“전체 코드 품질을 개선해줘” 같은 요청은 위험합니다. AI가 포맷, 구조, 이름, 로직을 한 번에 바꿀 수 있기 때문입니다.

좋은 요청은 “로그인 실패 시 사용자 메시지만 수정해줘”, “이 테스트 실패 원인만 찾아줘”처럼 범위가 좁습니다. 좋은 AI 요청은 좋은 커밋 단위와 비슷합니다.

### 21. AI 요청 원칙

Git 관점에서 좋은 AI 요청은 좋은 커밋 단위와 비슷합니다.  
하나의 목적, 명확한 범위, 검증 가능한 결과가 있어야 합니다.

AI에게 요청할 때는 “어떤 파일을 중심으로 봐야 하는지”, “어떤 동작은 유지해야 하는지”, “어떤 테스트를 실행해야 하는지”를 같이 알려주면 결과가 안정적입니다.

### 22. AI 결과 검토 루틴

AI 결과 검토는 두 단계로 나눌 수 있습니다.  
첫째는 사람의 검토입니다. diff를 읽고 요구사항과 맞는지 봅니다.  
둘째는 자동화 검증입니다. 테스트, 린트, 타입 체크를 실행합니다.

둘 중 하나만으로는 부족합니다. 테스트가 통과해도 요구사항이 틀릴 수 있고, 사람이 보기에는 맞아 보여도 숨은 회귀가 있을 수 있습니다.

### 23. 검증 명령 예시

눈으로 보고, 자동화로 다시 확인해야 합니다.  
프로젝트마다 명령은 다르지만 보통 테스트, 린트, 타입 체크, 빌드가 검증의 기본 축입니다.

AI에게도 “수정 후 테스트를 실행하고 결과를 요약해줘”라고 요청하는 것이 좋습니다. 다만 AI가 실행한 명령과 결과는 발표자가 아니라 실제 리뷰어 입장에서도 확인 가능해야 합니다.

### 24. 롤백

롤백이 쉬워야 실험이 쉬워집니다.  
AI를 쓰면 실험 속도가 빨라집니다. 그런데 되돌리기 어려운 방식으로 작업하면 실험이 부담이 됩니다.

작은 커밋, 명확한 브랜치, 깨끗한 작업 상태가 있으면 문제가 생겼을 때 특정 변경만 되돌릴 수 있습니다. 이것이 Git을 잘 쓰는 팀이 AI 도입에 더 유리한 이유입니다.

### 25. git revert

초보자에게 안전한 기본값은 `git revert`입니다.  
`revert`는 공유된 히스토리를 지우지 않고, 기존 커밋을 취소하는 새 커밋을 만듭니다.

이미 팀원과 공유된 브랜치에서는 히스토리를 강제로 바꾸는 방식보다 revert가 안전합니다. 중요한 것은 “되돌렸다”는 사실도 히스토리에 남는다는 점입니다.

### 26. 브랜치 전략

브랜치는 실험 공간입니다. main은 안정된 공간으로 두고, AI에게 맡기는 작업은 별도 브랜치에서 진행하는 것이 좋습니다.

특히 AI가 여러 파일을 수정하거나 구조를 바꾸는 작업이라면 브랜치를 분리해야 합니다. 그래야 실패해도 main에 영향을 주지 않고, 필요하면 브랜치 전체를 버릴 수 있습니다.

### 27. 브랜치 이름 예시

브랜치 이름은 작업의 목적이 드러나야 합니다.  
`feature/login-error-message`, `fix/payment-timeout`, `refactor/user-service`처럼 읽는 사람이 대략적인 변경 내용을 알 수 있으면 좋습니다.

AI 작업용 브랜치라고 해서 특별할 필요는 없습니다. 오히려 팀의 기존 브랜치 규칙을 그대로 따르는 것이 중요합니다.

### 28. 브랜치 생성

브랜치는 VSCode 상태바에서 만들 수도 있고 명령어로 만들 수도 있습니다.  
중요한 것은 어떻게 만드느냐보다, 작업을 시작하기 전에 분리한다는 점입니다.

AI에게 “새 브랜치에서 작업해줘”라고 지시할 수도 있지만, 실제로 어떤 브랜치에 있는지는 개발자가 확인해야 합니다. `git branch`나 VSCode 상태바를 보는 습관이 필요합니다.

### 29. Pull Request

PR은 리뷰 요청이자 설명 문서입니다.  
AI가 코드를 만들었더라도 PR을 읽는 사람은 사람입니다. 따라서 PR에는 무엇을 바꿨는지, 왜 바꿨는지, 어떻게 검증했는지가 있어야 합니다.

좋은 PR은 리뷰어의 시간을 줄여 줍니다. 리뷰어가 봐야 할 부분을 먼저 알려주고, 의도하지 않은 변경이 없다는 신뢰를 만들어 줍니다.

### 30. PR 설명 예시

PR 설명에는 보통 변경 요약, 검증 결과, 리뷰 포인트가 들어갑니다.  
예를 들어 “로그인 실패 시 서버 오류 메시지가 노출되던 문제를 사용자 친화적인 메시지로 변경했습니다. 관련 테스트를 추가했고, 기존 성공 로그인 흐름은 유지했습니다.”처럼 쓰면 리뷰어가 방향을 이해하기 쉽습니다.

AI에게 초안을 만들게 할 수 있지만, 마지막 사실 확인은 반드시 사람이 해야 합니다.

### 31. AI로 PR 설명 만들기

AI는 PR 설명 초안을 잘 만들어 줍니다. 하지만 AI가 쓴 설명도 사실 확인이 필요합니다.  
실제로 바뀌지 않은 내용을 쓸 수도 있고, 테스트하지 않은 항목을 테스트했다고 표현할 수도 있습니다.

따라서 AI에게는 “현재 git diff를 기준으로”라고 명확히 말하고, 나온 결과를 diff와 비교해야 합니다.

### 32. 좋은 요청 문장

좋은 요청 문장은 이렇게 만들 수 있습니다.  
“현재 git diff를 기준으로 PR 설명 초안을 작성해줘. 변경 요약, 테스트 결과, 리뷰어가 집중해서 볼 부분을 나눠서 작성해줘.”

여기서 핵심은 기준을 현재 diff로 제한하는 것입니다. AI가 추측으로 내용을 만들지 않도록 입력 범위를 좁혀야 합니다.

### 33. 실무 팁

실무에서 가장 중요한 팁은 AI가 만든 결과를 한 번에 커밋하지 않는 것입니다.  
먼저 diff를 읽고, 불필요한 변경을 제거하고, 의미 단위로 stage하고, 검증 결과를 남겨야 합니다.

이 루틴은 처음에는 조금 느려 보일 수 있지만, 장기적으로는 리뷰 시간과 장애 대응 시간을 줄여 줍니다.

### 34. 좋은 커밋 단위

좋은 커밋 단위를 판단하는 질문은 하나입니다. “나중에 이 커밋만 되돌려도 의미가 있는가?”  
만약 하나의 커밋 안에 버그 수정, 포맷 변경, 파일 구조 변경, 문서 수정이 모두 들어 있다면 되돌리기 어렵습니다.

AI가 한 번에 많은 변경을 만들수록 커밋 단위를 사람이 다시 나눠야 합니다.

### 35. 피해야 할 커밋

피해야 할 커밋은 세 가지입니다.  
첫째, 너무 큰 커밋입니다. 리뷰어가 의도를 파악하기 어렵습니다.  
둘째, 여러 목적이 섞인 커밋입니다. 나중에 되돌리기 어렵습니다.  
셋째, 메시지가 의미 없는 커밋입니다. 히스토리를 검색할 때 도움이 되지 않습니다.

AI 시대에는 이런 커밋이 더 쉽게 생깁니다. 그래서 stage와 commit 메시지가 더 중요합니다.

### 36. 커밋 메시지 패턴

커밋 메시지는 팀 규칙에 맞추면 됩니다. 일반적으로 `fix:`, `feat:`, `refactor:`, `docs:` 같은 접두사를 사용하면 변경 성격이 잘 드러납니다.

예를 들어 `fix: 로그인 실패 메시지 노출 방식 수정`은 어떤 문제를 고쳤는지 알 수 있습니다. 반면 `update files`는 아무 정보도 주지 않습니다.

### 37. 충돌 해결

충돌은 실패가 아니라 대화입니다.  
Git 충돌은 같은 부분을 서로 다르게 바꿨기 때문에 사람이 결정해야 한다는 신호입니다.

AI에게 충돌 해결을 도와달라고 할 수는 있습니다. 하지만 최종적으로 어떤 변경을 살릴지 결정하는 것은 개발자입니다. 충돌 해결 후에는 반드시 테스트를 실행해야 합니다.

### 38. .gitignore와 비밀정보

커밋 전에는 새 파일을 더 조심해서 봐야 합니다.  
특히 `.env`, 인증키, 로컬 설정 파일, 로그 파일이 새 파일로 잡히는 경우가 있습니다. AI가 작업 중 생성한 임시 파일도 들어갈 수 있습니다.

`.gitignore`는 비밀정보를 보호하는 최소한의 방어선입니다. 하지만 `.gitignore`만 믿지 말고 커밋 전 파일 목록을 직접 확인해야 합니다.

### 39. 히스토리는 팀의 기억

좋은 Git 히스토리는 질문에 답합니다.  
이 기능은 언제 들어왔는가? 왜 이 예외 처리를 추가했는가? 문제가 생기면 어떤 커밋을 되돌려야 하는가?

AI가 코드를 많이 만들어도 히스토리가 잘 남아 있으면 팀은 변경을 이해할 수 있습니다. 반대로 히스토리가 엉켜 있으면 AI를 쓴 속도보다 나중에 해석하는 비용이 더 커집니다.

### 40. 마무리 체크리스트

마지막으로 체크리스트를 정리하겠습니다.  
AI 작업 전에는 `git status`로 깨끗한 상태인지 확인합니다.  
AI 작업 후에는 Source Control에서 파일 상태를 확인합니다.  
diff를 읽고, 불필요한 변경은 제거합니다.  
stage로 커밋 범위를 고르고, 의미 있는 메시지로 commit합니다.  
PR에는 변경 요약, 검증 결과, 리뷰 포인트를 남깁니다.

오늘의 핵심은 간단합니다. AI가 코드를 빨리 만들수록, 우리는 Git으로 변경을 더 잘 설명해야 합니다. 감사합니다.
