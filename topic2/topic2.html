Git의 역할 1

Git은 백업 도구가 아니다

GitHub에 올리는 저장소가 전부가 아닙니다. Git의 진짜 가치는 “왜 이렇게 바뀌었는지”를 남기는 데 있습니다.

백업은 최종 파일을 보존하지만, Git은 변경의 과정과 이유를 보존합니다.

AI가 만든 결과를 바로 믿기보다, 변경 단위로 읽고 선택할 수 있어야 합니다.

01 변경의 기록 최종 파일만 보존하는 것이 아니라, 코드가 언제 왜 바뀌었는지 과정까지 남깁니다.
main: 안정 상태 + 로그인 실패 메시지 표시
02 실험의 경계 AI에게 큰 변경을 맡겨도 브랜치와 커밋이 있으면 실패를 관리할 수 있습니다.
checkpoint - 원치 않는 변경은 되돌림
03 협업의 언어 말로 “대충 고쳤다”가 아니라 diff, commit, PR로 어떤 파일의 어떤 의도가 바뀌었는지 설명합니다. AI 시대에는 이 설명 능력이 코드 작성 능력만큼 중요합니다.
Git이 관리하는 세 가지
변경의 기록언제, 왜, 어떻게 바뀌었는지 남깁니다.
실험의 경계실패해도 돌아갈 수 있는 지점을 만듭니다.
협업의 언어말이 아니라 diff와 PR로 변경을 설명합니다.

Git을 잘 쓴다는 것은 저장소에 코드를 올리는 데서 끝나지 않습니다. 작업의 의도를 작게 나누고, 그 단위를 팀이 읽을 수 있게 남기는 것입니다.

AI 시대의 변화

코드 작성 속도보다
변경 검토 속도가 중요해진다

AI는 많은 코드를 빠르게 제안합니다. 개발자는 그 변경이 맞는지 판단해야 합니다.

예전에는 사람이 천천히 고친 코드만 검토하면 됐지만, 이제는 AI가 만든 대량 변경을 짧은 시간 안에 판별해야 합니다.

그래서 작업의 중심은 “많이 작성하기”에서 “잘 나누고, 잘 읽고, 잘 되돌리기”로 이동합니다.

겉으로 멀쩡해 보이는 코드의 위험
  • 기존 비즈니스 규칙을 놓쳤을 수 있습니다.
  • 테스트는 통과하지만 실제 사용 흐름에서 깨질 수 있습니다.
  • 보안상 위험한 처리를 넣었을 수 있습니다.
  • 프로젝트 스타일과 다른 방식을 섞었을 수 있습니다.
  • 필요 이상으로 많은 파일을 수정했을 수 있습니다.
검토 비용의 이동AI가 작성 시간을 줄여도, 변경을 이해하고 검증하는 시간은 사라지지 않습니다. 오히려 변경량이 늘면 더 중요해집니다.
판단 기준“코드가 그럴듯한가”보다 “요구사항, 예외 처리, 보안, 테스트, 프로젝트 관례를 지켰는가”를 봐야 합니다.
오늘의 네 가지 원칙
작게 바꾸고 자주 커밋한다검토 가능한 단위를 유지합니다.
AI 변경은 diff로 확인한다결과물이 아니라 제안된 패치로 봅니다.
실험은 브랜치에서 한다main은 안정적인 공간으로 남깁니다.
커밋 메시지는 설명이다미래의 나와 팀원을 위해 씁니다.

이 네 가지는 서로 연결됩니다. 작은 요청은 작은 diff를 만들고, 작은 diff는 좋은 커밋이 되며, 좋은 커밋은 리뷰 가능한 PR로 이어집니다.

VSCode Git 기본기

Source Control 패널은
매일 보는 대시보드다

AI 작업 후에는 가장 먼저 변경된 파일 목록을 확인합니다.

파일 상태 목록은 “AI가 어디까지 손댔는지” 보여주는 첫 번째 신호입니다.

의도하지 않은 설정 파일, 새 파일, 삭제 파일이 보이면 커밋 전에 반드시 멈춰야 합니다.

파일 상태 읽기

Source Control의 글자는 변경 상태를 알려주는 표지판입니다

M: ModifiedGit이 이미 추적하던 파일의 내용이 바뀐 상태입니다. diff를 열어 의도한 줄만 바뀌었는지 확인합니다.
U: UntrackedGit이 아직 추적하지 않는 새 파일입니다. `.env`, 임시 로그, 로컬 설정처럼 커밋하면 안 되는 파일이 섞일 수 있습니다.
A: Added새 파일이 이번 커밋 후보로 stage된 상태입니다. 정말 저장소에 들어가야 하는 새 파일인지 확인합니다.
D: Deleted추적 중이던 파일이 삭제된 상태입니다. AI가 테스트, 설정, 예외 처리 파일을 지운 것은 아닌지 반드시 봅니다.

핵심은 상태 글자를 외우는 것이 아니라, 각 상태가 “커밋 전 확인해야 할 질문”을 만든다는 점입니다.

Diff 보기

AI 결과를 믿기 전에
변경 자체를 읽는다

초록색 추가만 보지 말고, 빨간색 삭제를 더 조심해서 봅니다.

AI가 만든 코드는 “완성품”이 아니라 “제안된 패치”입니다.

패치는 적용하기 전에 읽어야 하고, diff는 그 패치를 가장 정확하게 보여주는 화면입니다.

- if (!user.hasPermission) return deny()
+ return allow()

+ showLoginError(message)
  trackLoginAttempt()
Diff에서 확인할 것
  • 요구사항과 관련 있는 변경인가?
  • 기존 로직을 실수로 지우지 않았는가?
  • 파일 위치와 함수 이름이 프로젝트 관례에 맞는가?
  • 에러 처리와 예외 상황이 빠지지 않았는가?
  • 테스트 코드도 함께 바뀌었는가?

특히 삭제된 코드는 추가된 코드보다 더 위험할 수 있습니다. 권한 체크, 예외 처리, 로그, 테스트가 사라졌다면 기능이 깔끔해진 것이 아니라 안전장치가 빠진 것일 수 있습니다.

Stage

Stage는 이번 커밋에 담을 변경을 고르는 장바구니다

AI가 여러 파일을 바꿨다면, 의미 있는 단위로 나누어 stage합니다.

Git의 commit은 작업 폴더 전체를 무조건 저장하는 것이 아니라, staging area에 올려둔 변경의 스냅샷을 기록합니다.

따라서 stage는 단순한 중간 단계가 아니라 “이번 커밋의 범위와 의미를 정하는 편집 과정”입니다.

로그인 버그요구사항과 직접 관련된 변경이므로 이번 커밋에 포함합니다.
README 문구동작 변경과 목적이 다르므로 문서 커밋으로 분리합니다.
임시 로그검토와 실행 중 필요했던 흔적은 커밋 전에 제거합니다.
Partial staging

한 파일 안에서도
일부 변경만 커밋할 수 있다

VSCode diff 화면에서 특정 변경 블록만 stage하면 히스토리가 훨씬 읽기 쉬워집니다.

AI 변경 분리AI가 로그인 코드와 README를 동시에 고쳤다면, 로그인 수정만 먼저 stage하고 문서 수정은 별도 커밋으로 분리합니다.
임시 작업 제외디버깅 로그, 실험 코드, 포맷팅 변경은 실제 수정 커밋에 섞이지 않게 제거하거나 따로 처리합니다.
Commit

Commit은 저장이 아니라 설명이다

나중에 문제가 생겼을 때 메시지만 봐도 의도를 알 수 있어야 합니다.

Git의 commit은 나중에 비교하거나 되돌릴 수 있는 프로젝트 스냅샷입니다.

메시지가 부실하면 스냅샷은 남아도 “왜 그 상태가 필요했는지”라는 맥락은 사라집니다.

전환점

이제 Git을
AI 작업의 안전장치로 써봅시다.

안전장치라는 말은 AI를 막는다는 뜻이 아닙니다. 변경 범위를 나누고, 확인하고, 필요하면 되돌릴 수 있게 만드는 작업 방식입니다.

AI 작업 전 체크포인트

먼저 현재 상태를 깨끗하게 만든다

git status

이미 작업 중인 변경이 있다면 먼저 커밋하거나 stash하거나 별도 브랜치로 옮깁니다.

내가 직접 만든 변경과 AI가 만든 변경이 섞이면, 나중에 어떤 판단으로 들어간 코드인지 구분하기 어렵습니다.

AI에게 일을 맡기기 전의 깨끗한 상태는 되돌아올 수 있는 출발점입니다.

좋은 흐름
01상태 확인현재 변경 파일을 확인하고, 이미 작업 중인 내용은 먼저 정리합니다.
02브랜치 생성AI 실험과 main의 안정 상태를 분리합니다.
03AI 요청하나의 목적만 가진 작은 요청으로 범위를 제한합니다.
04Diff 검토파일 목록, 삭제 코드, 테스트를 확인한 뒤 커밋합니다.

이 순서는 AI 도구와 상관없이 반복 가능한 체크리스트입니다. 시작점을 깨끗하게 만들고, 실험 공간을 나누고, 결과를 diff로 읽은 뒤에만 히스토리에 남깁니다.

요청 범위

AI에게 한 번에 너무 많이 맡기지 않는다

요청이 커질수록 변경 범위가 커지고, 검토 비용도 커집니다.

좋은 AI 요청은 좋은 커밋처럼 하나의 목적을 가져야 합니다.

“전체 개선”보다 “로그인 실패 메시지만 수정”이 검토하기 쉽고 되돌리기도 쉽습니다.

나쁜 요청이 프로젝트 리팩터링해줘.
좋은 요청UserService의 validation 로직만 별도 함수로 분리해줘. 동작은 바꾸지 마.
AI 요청 원칙

Git 관점에서 좋은 AI 요청은 좋은 커밋 단위와 비슷합니다.

하나의 요청은 하나의 목적을 가져야 합니다.

범위 제한수정 파일, 관련 테스트, 바꾸지 말아야 할 동작을 함께 말하면 검토할 diff가 작아집니다.
검토 가능성한 번에 많은 것을 맡기면 편해 보이지만, 실제로는 리뷰 비용과 롤백 비용이 커집니다.
AI 결과 검토 루틴
파일 목록의도하지 않은 파일이 바뀌었는지 확인합니다. 설정, 빌드, lock 파일 변경은 특히 이유를 확인합니다.
삭제된 코드권한 체크, 예외 처리, 로그, 테스트가 사라졌는지 봅니다. 삭제는 추가보다 더 조용히 문제를 만듭니다.
테스트구현에 맞춘 테스트인지, 실제 요구사항을 검증하는 테스트인지 봅니다. 없는 경우 수동 확인 범위를 넓힙니다.
직접 실행주요 화면과 흐름을 한 번 확인합니다. 자동화가 통과해도 실제 사용 흐름에서 깨질 수 있습니다.

VSCode의 Source Control과 diff 화면은 이 루틴을 한 곳에서 진행하기 위한 작업대입니다.

실험이 쉬워진다

실패해도 되돌릴 수 있으면 AI 실험을 더 안전하게 할 수 있습니다.

AI에게 맡긴 작업이 기대와 다르면, 설득하려고 오래 붙잡기보다 변경 단위를 되돌리고 더 작은 요청으로 다시 시작할 수 있어야 합니다.

restore, revert, reset은 모두 되돌리는 명령처럼 보이지만 적용 대상과 위험도가 다릅니다.

restore커밋 전 작업 폴더의 파일 변경을 되돌릴 때 씁니다. 아직 히스토리에 남지 않은 실험을 버리는 용도입니다.
revert이미 기록된 커밋을 취소하는 새 커밋을 만듭니다. 공유된 브랜치에서 가장 설명하기 쉬운 기본값입니다.
reset브랜치가 가리키는 위치를 옮깁니다. 개인 작업 정리에는 유용하지만 공유 브랜치에서는 매우 조심해야 합니다.
초보자에게 안전한 기본값

git revert는 공유된 히스토리를 망가뜨리지 않고 되돌립니다.

restore아직 커밋하지 않은 파일 변경을 되돌릴 때 자주 사용합니다.
revert이미 공유된 커밋을 취소하는 새 커밋을 만들어, 협업 히스토리를 안전하게 유지합니다.
브랜치 전략

브랜치는
실험실이다.

main은 안정적인 코드가 있는 공간으로 두고, AI 실험이나 새 기능은 별도 브랜치에서 진행합니다.

main팀이 기준으로 삼는 안정 상태입니다. 바로 실험하면 롤백과 리뷰 범위가 흐려집니다.
작업 브랜치기능, 버그 수정, AI 실험을 분리하는 임시 작업 공간입니다. 실패하면 브랜치 단위로 버릴 수 있습니다.
브랜치 이름 예시

main은 안정 공간,
실험은 별도 브랜치

feature/login-error-message
fix/session-redirect
refactor/user-validation
ai/prototype-search-ui
main 배포 가능하거나 팀이 기준으로 삼는 안정 브랜치입니다.
feature / fix 기능 추가, 버그 수정처럼 리뷰 가능한 작업 단위를 분리합니다.
ai / prototype AI로 실험한 변경을 명확히 구분하고, 필요하면 쉽게 버립니다.
PR review 실험이 끝나면 diff와 테스트 결과를 설명하는 리뷰 문서로 전환합니다.
1
main에서 바로 실험하지 않기안정 상태를 기준점으로 두고 새 브랜치를 만듭니다.
2
feature / fix로 목적 분리기능 추가, 버그 수정, 리팩터링을 서로 다른 리뷰 단위로 나눕니다.
3
ai / prototype으로 실험 표시AI가 만든 실험성 변경은 브랜치 이름만 봐도 구분되게 합니다.
4
PR에서 설명과 검토로 전환diff, 테스트 결과, 리뷰 포인트를 남겨 협업 가능한 형태로 만듭니다.
브랜치 생성

VSCode 상태바에서 만들 수도 있고,
명령어로 만들 수도 있습니다.

git switch -c feature/login-error-message

브랜치는 거창한 기능이 아니라 작업 공간을 분리하는 방법입니다. AI에게 실험적인 수정을 맡길 때 main에서 바로 하지 않는 것만으로도 검토와 롤백이 쉬워집니다.

Pull Request

PR은 리뷰 요청이자 설명 문서다

무엇을변경했는가. 어떤 파일과 흐름이 영향을 받는가.
왜버그 수정인지, 기능 추가인지, 리팩터링인지 이유를 남깁니다.
어떻게테스트했는가. 미확인 항목은 미확인으로 남깁니다.
PR의 본질PR은 main에 합치는 버튼이 아니라, merge 전에 변경을 논의하고 리뷰하는 협업 공간입니다.
AI 변경일수록 중요AI가 만든 코드는 작성자가 의도를 직접 경험하지 않았을 수 있으므로, diff 기반 설명과 리뷰 포인트가 더 필요합니다.
PR 설명 예시

리뷰어가 볼 부분을
먼저 알려준다

좋은 PR 설명은 리뷰어가 전체 코드를 다시 읽지 않아도 변경의 목적과 위험 지점을 빠르게 파악하게 합니다.

테스트 결과, 수동 확인, 미확인 항목을 분리해서 쓰면 리뷰 품질이 좋아집니다.

AI로 PR 설명 만들기

AI가 쓴 설명도 사실 확인이 필요하다

AI는 실행하지 않은 테스트를 실행했다고 쓰거나, 실제로 바꾸지 않은 내용을 바꿨다고 말할 수 있습니다.

diff 기반으로 제한PR 설명은 현재 diff에서 확인되는 내용만 쓰도록 요청해야 합니다.
미확인은 미확인실제로 실행하지 않은 테스트는 “미확인”으로 남겨야 리뷰어가 판단할 수 있습니다.
좋은 요청 문장 
현재 git diff를 기준으로 PR 설명 초안을 작성해줘.
실제로 확인되지 않은 테스트는 "미확인"으로 표시해줘.
추측하지 말고 diff에서 확인되는 내용만 써줘.

AI를 문서화 보조 도구로 쓰는 것은 좋지만, 최종 PR 설명의 책임은 사람에게 있습니다.

실무 팁

좋은 히스토리는
나중의 디버깅 비용을 줄인다.

히스토리는 단순 로그가 아니라 팀의 기억입니다. 문제를 추적할 때 “왜 이 코드가 들어왔는지”를 알려주는 근거가 됩니다.

좋은 커밋 단위

나중에 이 커밋만 되돌려도 의미가 있는가?

커밋 단위는 작게 쪼갠 작업의 결과이면서, 나중에 문제를 찾는 검색 단위입니다.

AI가 여러 일을 한 번에 고쳤다면 사람이 stage를 나누어 커밋 단위를 다시 만들어야 합니다.

  • 로그인 실패 메시지 표시 버그 수정
  • 사용자 입력 validation 함수 분리
  • 검색 결과 empty state UI 추가
  • 결제 API timeout 테스트 추가
피해야 할 커밋
섞인 커밋여러 기능, 리팩터링, 포맷팅이 한 번에 들어감
목적이 다른 커밋동작 변경과 코드 정리가 섞임
임시 작업 포함임시 로그와 실제 수정이 함께 들어감

AI는 종종 “겸사겸사” 많은 것을 고칩니다. 사람이 커밋 단위를 나누지 않으면 히스토리가 빠르게 흐려집니다.

커밋 메시지 패턴

type: summary

type보다 중요한 것은 구체적인 summary입니다.

`fix: bug`처럼 넓은 표현은 나중에 아무 도움이 되지 않습니다.

`fix: prevent duplicate payment submission`처럼 문제와 의도가 보이게 써야 합니다.

fix: 만료 세션 리다이렉트 오류 수정
feat: 검색 결과 없음 화면 추가
refactor: 사용자 검증 로직 분리
test: 결제 타임아웃 테스트 추가
docs: 개발 환경 설정 문서 업데이트
충돌 해결

충돌은 실패가 아니라 대화다

충돌 해결은 “둘 중 하나 고르기”가 아니라 최종적으로 맞는 코드를 다시 구성하는 일입니다.

Accept Current Change Accept Incoming Change Accept Both Changes Compare Changes
버튼을 기계적으로 누르지 않기current와 incoming 중 하나가 항상 정답은 아닙니다. 두 변경을 읽고 최종 코드를 다시 구성합니다.
AI에게 맡겨도 최종 diff 확인충돌 해결 과정에서 비즈니스 로직, 권한 체크, 테스트가 사라질 수 있습니다.
.gitignore와 비밀정보

커밋 전에는 새 파일을 더 조심해서 본다

AI가 임시 키나 설정 파일을 만들 수 있고, 사용자가 모르고 커밋할 수 있습니다.

`.gitignore`는 Git이 의도적으로 추적하지 않을 파일을 지정하는 장치입니다.

팀과 공유해야 하는 ignore 규칙은 저장소 루트의 `.gitignore`에 두고, 개인 도구가 만든 파일은 개인 제외 설정으로 분리하는 편이 좋습니다.

  • .env 같은 환경변수 파일
  • API key, token, password
  • 빌드 결과물과 로컬 캐시
  • 공유할 필요 없는 개인 IDE 설정
히스토리는 팀의 기억

좋은 Git 히스토리는 질문에 답합니다

언제 들어왔는가?기능이나 예외 처리가 처음 추가된 시점을 찾고, 관련 PR과 요구사항으로 이어갑니다.
왜 필요한가?커밋 메시지와 PR 설명을 통해 당시의 제약, 버그, 결정 이유를 이해합니다.
무슨 버그였는가?같은 문제가 반복될 때 과거 수정 방향과 놓친 테스트를 함께 확인할 수 있습니다.
어디부터 봐야 하는가?문제가 생겼을 때 전체 코드를 뒤지지 않고 의심 커밋과 변경 범위를 좁힙니다.

AI가 코드를 많이 만들어도, 맥락을 보존하는 일은 여전히 사람의 책임입니다.

오늘 내용을 하나로 줄이면 “AI가 만든 결과를 바로 믿지 말고, Git으로 변경을 읽고 선택하자”입니다. Git diff를 읽는 습관이 사고를 줄입니다.

결론: AI가 코드를 바꾼 뒤에는 반드시 Git diff를 읽자.
