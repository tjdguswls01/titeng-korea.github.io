Git Command Prompt Guide

Git 명령어 20가지
명령어에서 프롬프트로

예전에는 옵션과 순서를 직접 외워 실행했습니다. 이제는 목표, 범위, 안전 조건, 검증 기준을 AI 에이전트에게 전달하고 결과를 확인합니다.

사용 방식의 변화

명령어를 없애는 것이 아니라
작업 의도를 더 정확히 전달하는 방식입니다.

Before명령어와 옵션을 직접 조합하고, 출력 결과를 읽은 뒤 다음 명령을 사람이 결정했습니다.
NowAI 에이전트에게 목표와 제약을 말하고, 에이전트가 실행한 명령과 diff, 테스트 결과를 사람이 검토합니다.
좋은 프롬프트무엇을 하려는지, 어떤 파일을 건드려도 되는지, 위험하면 어디서 멈출지를 포함합니다.
마지막 책임커밋, push, reset 같은 변경 작업은 에이전트 결과를 확인한 뒤 승인하는 흐름이 안전합니다.
01. 상태 확인

git status
지금 저장소가 어떤 상태인지 볼 때

AI 에이전트를 쓸 때도 첫 단계는 상태 확인입니다. 단순 출력보다 위험 파일과 다음 행동까지 요약하게 합니다.

기존 사용command
git status
git status -sb
수정, 삭제, untracked 파일을 직접 읽고 다음 명령을 판단했습니다.
AI 프롬프트prompt
현재 저장소 상태를 확인하고, 수정/추가/삭제/추적 제외 파일을 표로 요약해줘. 커밋에 포함하면 안 될 파일이 있으면 표시해줘.
확인: 에이전트가 실행한 명령과 untracked 파일 목록을 검토합니다.
02. 변경 검토

git diff
커밋 전 변경 내용을 확인할 때

diff는 AI 작업 결과를 검증하는 핵심입니다. 의도된 변경과 위험한 변경을 나눠 설명하게 하면 검토 속도가 빨라집니다.

기존 사용command
git diff
git diff --staged
git diff src/app.js
추가/삭제 라인을 직접 훑으며 실수나 보안 정보를 찾았습니다.
AI 프롬프트prompt
아직 커밋하지 않은 변경을 검토해줘. 의도된 변경, 위험한 변경, 확인 질문으로 나눠 요약하고 테스트나 설정 삭제가 있으면 강조해줘.
확인: 삭제된 코드, 설정 파일, 테스트 변경을 직접 다시 봅니다.
03. 변경 선택

git add
이번 커밋에 넣을
파일을 고를 때

이제는 “무엇을 stage할지”보다 “이번 커밋의 목적이 무엇인지”를 먼저 알려주는 편이 좋습니다.

기존 사용command
git add src/app.js
git add .
git add -p
파일 단위 또는 hunk 단위로 직접 stage했습니다.
AI 프롬프트prompt
이번 커밋 목적은 로그인 오류 문구 수정이야. 관련 변경만 stage하고, 제외한 파일이 있으면 파일명과 제외 이유를 알려줘.
확인: `git diff --staged`로 stage된 범위를 확인합니다.
04. 커밋 작성

git commit
변경을 설명 가능한 기록으로 남길 때

커밋 메시지는 변경의 의도를 남기는 문서입니다. 에이전트에게 diff 기반 메시지를 만들게 하되 최종 문구는 확인합니다.

기존 사용command
git commit -m "fix: 로그인 실패 메시지 표시"
git commit
직접 메시지를 작성하고 커밋했습니다.
AI 프롬프트prompt
staged 변경을 읽고 Conventional Commit 형식으로 한국어 커밋 메시지 3개를 추천해줘. 가장 적절한 하나를 선택해 커밋해줘.
확인: 커밋 전 staged diff와 메시지가 같은 의도인지 봅니다.
05. 브랜치 확인

git branch
작업 위치와 브랜치 목록을 볼 때

브랜치 명령은 현재 위치를 확인하는 용도에서, 작업 전략을 점검하는 프롬프트로 확장됩니다.

기존 사용command
git branch
git branch -a
git branch --show-current
현재 브랜치와 로컬/원격 브랜치를 직접 확인했습니다.
AI 프롬프트prompt
현재 브랜치와 로컬/원격 브랜치 목록을 확인하고, 지금 작업에 새 브랜치가 필요한지 제안해줘.
확인: 에이전트가 제안한 기준 브랜치가 올바른지 확인합니다.
06. 브랜치 이동

git switch
작업 브랜치로 이동하거나 새로 만들 때

이동 전 미커밋 변경이 있으면 작업을 잃을 수 있습니다. 프롬프트에 멈춤 조건을 넣습니다.

기존 사용command
git switch main
git switch -c feature/login-message
직접 브랜치를 만들고 이동했습니다.
AI 프롬프트prompt
main 최신 상태에서 feature/login-message 브랜치를 만들고 이동해줘. 이동 전 미커밋 변경이 있으면 멈추고 먼저 알려줘.
확인: 현재 브랜치명과 기준 브랜치를 다시 확인합니다.
07. 이전 상태 참조

git checkout
특정 커밋이나 파일 상태를 볼 때

checkout은 강력하지만 실수하기 쉽습니다. 현재 작업을 보존한다는 조건을 명확히 줍니다.

기존 사용command
git checkout main
git checkout 3f4a9c2 -- src/app.js
브랜치 이동과 파일 복구를 같은 명령으로 처리했습니다.
AI 프롬프트prompt
이전 커밋의 파일 내용을 확인해야 해. 현재 작업을 보존하면서 src/app.js만 특정 커밋 기준으로 비교하거나 복구하는 안전한 방법을 실행해줘.
확인: 작업 트리가 덮어써지지 않았는지 `git status`로 봅니다.
08. 원격 정보 가져오기

git fetch
원격 최신 상태만
가져올 때

fetch는 로컬 작업을 바꾸지 않고 원격 정보를 갱신합니다. 에이전트에게 뒤처진 정도까지 요약하게 합니다.

기존 사용command
git fetch
git fetch --prune
git branch -vv
원격 정보를 받은 뒤 브랜치별 ahead/behind를 직접 봤습니다.
AI 프롬프트prompt
원격 저장소 최신 브랜치와 태그 정보를 가져오고, 내 로컬 브랜치가 원격보다 앞서거나 뒤처졌는지 요약해줘.
확인: 로컬 파일 변경이 발생하지 않았는지 확인합니다.
09. 원격 변경 반영

git pull
원격 변경을 현재 브랜치에 가져올 때

pull은 fetch와 merge 또는 rebase를 함께 수행합니다. 충돌이 나면 자동 해결보다 보고를 먼저 요구합니다.

기존 사용command
git pull
git pull --rebase
원격 변경을 받고 충돌이 나면 직접 해결했습니다.
AI 프롬프트prompt
현재 브랜치에 원격 변경을 반영해줘. 충돌이 나면 자동 커밋하지 말고 충돌 파일과 해결 방향을 먼저 보고해줘.
확인: pull 방식이 merge인지 rebase인지 확인합니다.
10. 원격 업로드

git push
내 커밋을
원격 저장소에 올릴 때

push 전에는 브랜치명, 커밋 범위, 원격 주소를 확인해야 합니다. 처음 올릴 때는 upstream 설정도 필요합니다.

기존 사용command
git push
git push -u origin feature/login-message
직접 원격 브랜치에 업로드했습니다.
AI 프롬프트prompt
현재 브랜치를 origin에 올려줘. upstream이 없으면 설정하고, 푸시 전 마지막 커밋과 브랜치명을 확인해줘.
확인: 원격에 올릴 브랜치와 커밋 수를 확인합니다.
11. 기록 확인

git log
커밋 히스토리를 읽을 때

히스토리는 단순 목록보다 맥락이 중요합니다. 에이전트에게 변경 흐름과 의심 커밋을 요약하게 합니다.

기존 사용command
git log
git log --oneline --graph --decorate --all
커밋 그래프를 보고 문제 커밋을 직접 추적했습니다.
AI 프롬프트prompt
최근 커밋 히스토리를 기능 단위로 요약하고, 이번 문제와 관련 있을 만한 커밋 후보를 근거와 함께 찾아줘.
확인: 제안된 커밋의 diff와 날짜를 직접 확인합니다.
12. 변경 되돌리기

git restore
커밋 전 변경을
파일 단위로 버릴 때

restore는 작업 내용을 삭제할 수 있습니다. 에이전트에게 먼저 diff를 확인하고 판단 근거를 말하게 합니다.

기존 사용command
git restore src/app.js
git restore --staged src/app.js
파일 변경을 직접 되돌리거나 stage에서 내렸습니다.
AI 프롬프트prompt
커밋하지 않은 변경 중 src/app.js만 되돌리고 싶어. 먼저 diff를 보여주고, 되돌려도 되는지 판단 근거를 알려줘.
확인: 되돌리기 전 필요한 변경이 남아 있지 않은지 봅니다.
13. 브랜치 병합

git merge
다른 브랜치의 변경을 합칠 때

merge 전에는 두 브랜치의 차이와 충돌 가능성을 확인합니다. 에이전트에게 병합 전 점검을 맡깁니다.

기존 사용command
git switch main
git merge feature/login-message
대상 브랜치로 이동해 직접 병합했습니다.
AI 프롬프트prompt
feature/login-message를 현재 브랜치에 병합해줘. 병합 전 두 브랜치 차이와 충돌 가능성이 높은 파일을 요약해줘.
확인: 병합 커밋이 필요한 상황인지 확인합니다.
14. 기록 재정렬

git rebase
내 브랜치를 최신 기준 위로 올릴 때

rebase는 히스토리를 다시 쓰는 작업입니다. 충돌 단계마다 멈추고 설명하게 하는 프롬프트가 안전합니다.

기존 사용command
git fetch
git rebase origin/main
git rebase --continue
충돌을 직접 해결하며 rebase를 진행했습니다.
AI 프롬프트prompt
내 feature 브랜치를 최신 main 위로 rebase해줘. 충돌이 나면 단계별로 멈추고 해결 내용과 테스트 필요 지점을 알려줘.
확인: 이미 공유한 브랜치라면 force push 여부를 신중히 봅니다.
15. 임시 저장

git stash
작업 중 변경을
잠시 치워둘 때

stash는 편하지만 잊기 쉽습니다. 무엇을 저장했고 언제 다시 적용할지 설명하게 합니다.

기존 사용command
git stash push -m "wip: login"
git stash list
git stash pop
작업을 임시 저장하고 나중에 직접 복원했습니다.
AI 프롬프트prompt
지금 작업을 임시 저장하고 main으로 이동해야 해. 추적되지 않은 파일 포함 여부를 확인해서 안전하게 stash해줘.
확인: stash 메시지와 포함 파일 목록을 확인합니다.
16. 커밋 되돌림

git reset
커밋이나 stage 상태를 뒤로 돌릴 때

reset은 작업 손실 가능성이 큽니다. 실행보다 설명과 안전한 선택지가 먼저입니다.

기존 사용command
git reset HEAD~1
git reset --soft HEAD~1
git reset --hard HEAD~1
목표에 따라 soft, mixed, hard를 직접 선택했습니다.
AI 프롬프트prompt
마지막 커밋을 되돌리는 방법을 soft, mixed, hard 차이로 설명하고 내 목표에 맞는 가장 안전한 reset만 실행해줘. 작업 손실 가능성이 있으면 멈춰줘.
확인: hard reset은 백업 또는 명시 승인 전 실행하지 않습니다.
17. 원격 저장소 관리

git remote
연결된 원격 저장소를 확인할 때

remote는 협업 대상과 권한을 결정합니다. URL 변경은 영향이 크므로 점검과 제안을 분리합니다.

기존 사용command
git remote -v
git remote add origin URL
git remote set-url origin URL
원격 이름과 URL을 직접 관리했습니다.
AI 프롬프트prompt
연결된 원격 저장소를 확인하고 origin URL이 올바른지 점검해줘. 필요하면 SSH/HTTPS 변경 명령을 제안해줘.
확인: 실제 URL 변경은 권한과 저장소명을 확인한 뒤 진행합니다.
18. 저장소 복제

git clone
새 환경에 저장소를 가져올 때

clone 이후에는 실행 방법, 의존성, 브랜치 기준을 확인해야 합니다. 에이전트에게 초기 점검까지 맡깁니다.

기존 사용command
git clone git@github.com:org/repo.git
cd repo
복제 후 README를 직접 읽고 환경을 맞췄습니다.
AI 프롬프트prompt
이 저장소를 새 폴더에 복제하고 실행 방법을 확인해줘. README 기준으로 필요한 초기 설정과 확인 명령도 요약해줘.
확인: 복제 위치와 원격 URL이 의도한 저장소인지 봅니다.
19. 버전 태그

git tag
릴리스 지점을 표시할 때

태그는 배포 기준점입니다. 생성 전 변경 요약과 대상 커밋을 확인하는 프롬프트가 필요합니다.

기존 사용command
git tag v1.2.0
git tag -a v1.2.0 -m "release: v1.2.0"
git push origin v1.2.0
태그를 만들고 원격에 직접 올렸습니다.
AI 프롬프트prompt
이번 릴리스에 v1.2.0 태그를 만들 준비가 됐는지 확인하고, 변경 요약과 태그 생성/푸시 절차를 진행해줘.
확인: 태그 대상 커밋과 버전 번호가 맞는지 검토합니다.
20. 특정 커밋 적용

git cherry-pick
필요한 커밋만 가져올 때

cherry-pick은 특정 변경만 옮길 때 유용하지만 충돌과 중복 적용을 조심해야 합니다.

기존 사용command
git cherry-pick 3f4a9c2
git cherry-pick --abort
커밋 해시를 직접 골라 현재 브랜치에 적용했습니다.
AI 프롬프트prompt
hotfix 브랜치의 특정 커밋만 현재 브랜치에 적용해줘. 적용 전 변경 파일을 확인하고 충돌 시 멈춰서 보고해줘.
확인: 이미 포함된 변경인지, 테스트가 필요한 범위가 무엇인지 봅니다.
