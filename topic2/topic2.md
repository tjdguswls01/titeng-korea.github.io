## 발표 개요

- 주제: AI 시대에 Git을 어떻게 더 안전하고 빠르게 활용할 것인가
- 대상: Git을 기본적으로 쓰고 있거나, VSCode에서 Git을 더 잘 쓰고 싶은 개발자/학생/실무자
- 목표:
  - Git을 단순 백업 도구가 아니라 사고 기록과 협업 장치로 이해한다.

  - VSCode에서 실무적으로 자주 쓰는 Git 흐름을 익힌다.
  - AI가 만든 변경사항을 검토하고 커밋하는 습관을 만든다.

## 0:00-3:00 오프닝

### 슬라이드 1. 제목

안녕하세요. 오늘 발표 주제는 **"AI 시대의 Git 활용법, with VSCode"**입니다.

요즘 개발 환경이 정말 빠르게 바뀌고 있습니다. 예전에는 개발자가 직접 코드를 한 줄씩 작성하고, 검색하고, 문서를 읽고, 시행착오를 겪으면서 코드를 완성했습니다. 그런데 이제는 AI에게 "이 기능 만들어줘", "이 버그 고쳐줘", "이 코드 리팩터링해줘"라고 말하면 꽤 그럴듯한 결과가 바로 나옵니다.

그런데 여기서 중요한 질문이 생깁니다.

**AI가 코드를 더 빨리 만들 수 있게 되었는데, 우리는 그 코드를 어떻게 믿고, 어떻게 관리해야 할까요?**

오늘의 핵심 답은 Git입니다.

AI 시대에는 Git이 단순히 "코드 저장하는 도구"가 아닙니다. Git은 AI가 만든 변경사항을 검증하고, 비교하고, 되돌리고, 협업 가능한 형태로 정리하는 안전장치입니다.

오늘은 Git의 모든 명령어를 외우는 시간이 아닙니다. 대신 VSCode를 중심으로, 실제 작업에서 Git을 어떻게 써야 AI와 함께 일할 때 덜 위험하고 더 생산적인지 살펴보겠습니다.

## 3:00-9:00 Git의 역할 재정의

### 슬라이드 2. Git은 백업 도구가 아니다

먼저 Git에 대한 오해부터 정리해보겠습니다.

많은 분들이 Git을 처음 배울 때 이렇게 생각합니다.

"Git은 코드 백업하는 도구 아닌가요?"

물론 틀린 말은 아닙니다. GitHub에 올려두면 내 컴퓨터가 고장 나도 코드를 복구할 수 있습니다. 하지만 Git의 진짜 가치는 백업보다 훨씬 큽니다.

Git은 세 가지를 관리합니다.

첫 번째, **변경의 기록**입니다.

코드가 언제, 왜, 어떻게 바뀌었는지를 남깁니다. 단순히 최종 결과만 보는 것이 아니라, 그 결과에 도달한 과정을 볼 수 있게 합니다.

두 번째, **실험의 경계**입니다.

새 기능을 만들거나, 라이브러리를 바꾸거나, AI에게 대규모 리팩터링을 맡길 때 우리는 실패할 수도 있습니다. Git을 잘 쓰면 실패가 두렵지 않습니다. 언제든 돌아올 수 있는 지점이 있기 때문입니다.

세 번째, **협업의 언어**입니다.

"이 코드 괜찮나요?"라고 말로 설명하는 대신, Git diff와 pull request를 통해 어떤 파일의 어떤 줄이 바뀌었는지 보여줄 수 있습니다. 팀원은 전체 코드를 다시 읽는 것이 아니라 변경된 부분을 중심으로 리뷰할 수 있습니다.

### 슬라이드 3. AI 시대에는 변경량이 폭발한다

AI 도구를 쓰면 코드 작성 속도가 빨라집니다. 하지만 동시에 변경량도 커집니다.

예전에는 개발자가 30분 동안 20줄을 고쳤다면, 지금은 AI가 30초 만에 200줄을 고칠 수 있습니다.

이건 생산성일 수도 있지만, 위험일 수도 있습니다.

AI가 만든 코드는 겉으로는 자연스러워 보입니다. 변수명도 그럴듯하고, 주석도 친절하고, 구조도 깔끔해 보일 수 있습니다. 하지만 다음과 같은 문제가 숨어 있을 수 있습니다.

- 기존 비즈니스 규칙을 놓쳤을 수 있습니다.
- 테스트는 통과하지만 실제 사용 흐름에서 깨질 수 있습니다.
- 보안상 위험한 처리를 넣었을 수 있습니다.
- 프로젝트 스타일과 다른 방식을 섞었을 수 있습니다.
- 필요 이상으로 많은 파일을 수정했을 수 있습니다.

그래서 AI 시대의 개발자는 "코드를 많이 쓰는 사람"에서 "변경을 잘 검토하고 통제하는 사람"으로 역할이 조금 이동합니다.

이때 Git은 변경을 통제하는 가장 기본적인 도구입니다.

### 슬라이드 4. 오늘의 핵심 원칙

오늘 발표 전체를 관통하는 원칙은 네 가지입니다.

첫 번째, **작게 바꾸고 자주 커밋한다.**

두 번째, **AI가 만든 변경은 반드시 diff로 확인한다.**

세 번째, **실험은 브랜치에서 한다.**

네 번째, **커밋 메시지는 미래의 나와 팀원을 위한 설명이다.**

이 네 가지만 습관으로 만들어도 Git 활용 수준이 크게 올라갑니다.

## 9:00-17:00 VSCode Git 기본기

### 슬라이드 5. VSCode Source Control 패널

이제 VSCode 화면을 기준으로 Git을 보겠습니다.

VSCode 왼쪽 사이드바를 보면 Source Control 아이콘이 있습니다. 보통 가지가 갈라지는 모양의 아이콘입니다.

이 패널에서 가장 먼저 보이는 것은 변경된 파일 목록입니다.

파일 옆에 표시되는 상태를 보면 대략 이런 의미입니다.

- M: Modified, 기존 파일이 수정됨
- U: Untracked, Git이 아직 추적하지 않는 새 파일
- A: Added, 새 파일이 stage됨
- D: Deleted, 파일이 삭제됨

Git을 CLI로 쓰면 `git status`를 입력해서 확인하는 내용이 VSCode에서는 이 패널에 시각적으로 표시됩니다.

여기서 중요한 것은 "내가 무엇을 바꿨는지 항상 보는 습관"입니다.

AI에게 작업을 맡긴 뒤에는 특히 Source Control 패널을 먼저 열어야 합니다. 내가 의도한 파일만 바뀌었는지 확인해야 합니다.

### 슬라이드 6. Diff 보기

변경된 파일을 클릭하면 VSCode가 diff 화면을 보여줍니다.

왼쪽은 이전 코드, 오른쪽은 현재 코드입니다. 또는 설정에 따라 줄 단위로 추가와 삭제가 표시됩니다.

초록색은 추가된 코드, 빨간색은 삭제된 코드입니다.

여기서 우리는 단순히 "코드가 예쁜가"를 보는 것이 아니라 다음을 확인해야 합니다.

- 요구사항과 관련 있는 변경인가?
- 기존 로직을 실수로 지우지 않았는가?
- 함수 이름이나 파일 위치가 프로젝트 관례와 맞는가?
- 에러 처리나 예외 상황이 빠지지 않았는가?
- 테스트 코드도 함께 바뀌었는가?

AI가 만든 코드를 검토할 때 가장 중요한 화면이 바로 diff입니다.

저는 개인적으로 AI 도구를 쓸 때, 결과 코드를 바로 믿지 않습니다. 대신 "AI가 제안한 패치"라고 생각합니다. 패치는 적용하기 전에 읽어야 합니다.

### 슬라이드 7. Stage와 Commit

Git에는 stage라는 개념이 있습니다.

처음 Git을 배울 때 stage가 조금 헷갈립니다. 쉽게 말하면 stage는 "이번 커밋에 넣을 변경사항을 고르는 장바구니"입니다.

작업 폴더에는 여러 변경사항이 있을 수 있습니다. 하지만 그 모든 변경사항을 한 커밋에 넣을 필요는 없습니다.

예를 들어 이런 상황을 생각해보겠습니다.

- 로그인 버그를 고쳤습니다.
- README 문구도 조금 수정했습니다.
- 실험하다가 임시 로그도 넣었습니다.

이 세 가지를 한 커밋에 다 넣으면 나중에 히스토리가 지저분해집니다. 로그인 버그 수정 커밋에는 로그인 버그와 관련된 파일만 stage하는 것이 좋습니다.

VSCode에서는 파일 옆의 `+` 버튼을 누르면 해당 파일이 stage됩니다. 파일 전체가 아니라 일부 줄만 stage할 수도 있습니다. diff 화면에서 특정 변경 블록만 선택해서 stage할 수 있습니다.

이 기능은 정말 중요합니다. AI가 여러 파일을 한 번에 수정했을 때, 의미 있는 단위로 나눠 커밋할 수 있기 때문입니다.

### 슬라이드 8. Commit은 저장이 아니라 설명이다

커밋을 "저장 버튼"처럼 쓰는 분들이 많습니다.

예를 들어 커밋 메시지가 이렇게 남습니다.

```text
update
fix
수정
작업
```

당장은 괜찮아 보입니다. 하지만 일주일 뒤, 한 달 뒤에 문제가 생기면 이 메시지는 아무 도움도 되지 않습니다.

좋은 커밋 메시지는 "무엇을 왜 바꿨는지" 알려줍니다.

예를 들어 이렇게 바꿀 수 있습니다.

```text
fix login redirect after expired session
```

또는 한국어로도 충분히 좋습니다.

```text
만료된 세션에서 로그인 후 리다이렉트 오류 수정
```

중요한 것은 언어가 아니라 구체성입니다.

AI 시대에는 커밋 메시지가 더 중요해집니다. 왜냐하면 코드 변경량은 늘어나는데, 사람이 기억할 수 있는 양은 그대로이기 때문입니다.

## 17:00-25:00 AI와 함께 쓰는 Git 흐름

### 슬라이드 9. AI 작업 전 체크포인트 만들기

AI에게 코드를 맡기기 전에 가장 먼저 할 일은 현재 상태를 깨끗하게 만드는 것입니다.

작업을 시작하기 전에 다음을 확인합니다.

```bash
git status
```

VSCode에서는 Source Control 패널을 보면 됩니다.

만약 이미 작업 중인 변경사항이 있다면 먼저 커밋하거나, 임시로 stash하거나, 별도 브랜치로 옮기는 것이 좋습니다.

왜냐하면 AI가 만든 변경사항과 내가 직접 만든 변경사항이 섞이면 검토가 어려워지기 때문입니다.

좋은 흐름은 이렇습니다.

1. 현재 변경사항 확인
2. 필요한 경우 커밋
3. 새 브랜치 생성
4. AI에게 작업 요청
5. diff 확인
6. 테스트 실행
7. 의미 있는 단위로 stage
8. 커밋

이 흐름을 습관으로 만들면 AI를 훨씬 안전하게 쓸 수 있습니다.

### 슬라이드 10. AI에게 한 번에 너무 많이 맡기지 않기

AI에게 이렇게 요청하고 싶은 순간이 있습니다.

"전체 프로젝트 구조 개선하고, 버그도 고치고, UI도 예쁘게 바꾸고, 테스트도 추가해줘."

편하긴 하지만 위험합니다.

요청이 커질수록 변경 범위가 커지고, 변경 범위가 커질수록 검토 비용이 커집니다.

AI와 일할 때는 작업을 작게 쪼개는 것이 좋습니다.

나쁜 요청 예시는 이렇습니다.

```text
이 프로젝트 리팩터링해줘.
```

더 좋은 요청은 이렇습니다.

```text
UserService의 중복된 validation 로직만 별도 함수로 분리해줘.
동작은 바꾸지 말고, 관련 테스트가 있으면 함께 업데이트해줘.
```

또는 이렇게 말할 수 있습니다.

```text
로그인 실패 시 에러 메시지가 표시되지 않는 문제만 수정해줘.
수정 파일은 로그인 화면과 관련 테스트로 제한해줘.
```

Git 관점에서 좋은 AI 요청은 좋은 커밋 단위와 비슷합니다.

하나의 요청은 하나의 목적을 가져야 합니다.

### 슬라이드 11. AI 결과 검토 루틴

AI가 코드를 수정한 뒤에는 바로 실행하기 전에 diff를 봅니다.

검토 루틴은 다음과 같습니다.

첫 번째, 파일 목록을 봅니다.

의도하지 않은 파일이 바뀌었는지 확인합니다. 예를 들어 로그인 버그를 고치라고 했는데 설정 파일, 빌드 파일, unrelated 컴포넌트가 바뀌었다면 멈추고 이유를 확인해야 합니다.

두 번째, 삭제된 코드를 봅니다.

추가된 코드보다 삭제된 코드가 더 위험할 때가 많습니다. 기존에 있던 예외 처리, 권한 체크, 로그, 테스트가 사라졌는지 봐야 합니다.

세 번째, 테스트를 봅니다.

AI가 테스트를 추가했다면 테스트가 의미 있는지 봐야 합니다. 단순히 구현에 맞춘 테스트인지, 실제 요구사항을 검증하는 테스트인지 확인합니다.

네 번째, 직접 실행합니다.

테스트가 있어도 실제 화면이나 주요 흐름을 한 번 확인하는 것이 좋습니다. VSCode에서는 터미널을 열고 프로젝트에 맞는 명령을 실행합니다.

예를 들어 JavaScript 프로젝트라면:

```bash
npm test
npm run lint
npm run dev
```

Python 프로젝트라면:

```bash
pytest
ruff check .
```

명령어는 프로젝트마다 다르지만 원칙은 같습니다. AI가 만든 변경을 사람의 눈과 자동화된 검증으로 함께 확인합니다.

### 슬라이드 12. 롤백이 쉬워야 실험이 쉬워진다

AI를 잘 쓰려면 실험을 많이 해야 합니다.

그런데 실험을 많이 하려면 실패해도 괜찮아야 합니다.

Git을 잘 쓰면 실패가 무섭지 않습니다.

VSCode에서는 파일 단위로 변경을 되돌릴 수 있습니다. Source Control 패널에서 파일을 우클릭하고 변경 취소를 선택하면 해당 파일의 수정사항을 되돌릴 수 있습니다.

명령어로는 다음과 같은 개념이 있습니다.

```bash
git restore 파일명
```

이미 커밋까지 했다면 커밋을 되돌리는 방식도 있습니다.

```bash
git revert 커밋해시
```

여기서 주의할 점은 `restore`, `reset`, `revert`의 차이를 이해해야 한다는 것입니다.

- restore: 아직 커밋하지 않은 파일 변경을 되돌릴 때 자주 사용
- revert: 기존 커밋을 취소하는 새 커밋을 만들 때 사용
- reset: 히스토리 자체를 이동시키는 강한 명령이므로 협업 브랜치에서는 조심

초보자에게 추천하는 안전한 기본값은 `revert`입니다. 공유된 히스토리를 망가뜨리지 않고 되돌릴 수 있기 때문입니다.

## 25:00-33:00 브랜치와 PR 전략

### 슬라이드 13. 브랜치는 실험실이다

브랜치는 어렵게 생각할 필요가 없습니다.

브랜치는 "작업을 분리하는 공간"입니다.

main 브랜치는 안정적인 코드가 있는 곳이라고 생각하고, 새 기능이나 AI 실험은 별도 브랜치에서 진행합니다.

예를 들어 브랜치 이름은 이렇게 만들 수 있습니다.

```text
feature/login-error-message
fix/session-redirect
refactor/user-validation
ai/prototype-search-ui
```

여기서 `ai/` 같은 prefix를 쓰면 AI로 실험한 브랜치를 구분하기 쉽습니다. 꼭 정답은 아니지만 팀에서 합의하면 유용합니다.

VSCode에서는 왼쪽 아래 상태바에서 현재 브랜치를 볼 수 있고, 클릭하면 브랜치를 만들거나 전환할 수 있습니다.

명령어로는 다음과 같습니다.

```bash
git switch -c feature/login-error-message
```

이 명령은 새 브랜치를 만들고 그 브랜치로 이동합니다.

### 슬라이드 14. PR은 리뷰 요청이자 설명 문서다

GitHub, GitLab, Bitbucket 같은 플랫폼에서는 Pull Request 또는 Merge Request를 사용합니다.

PR은 단순히 "main에 합쳐주세요"라는 버튼이 아닙니다.

PR은 변경사항을 설명하고 리뷰받는 문서입니다.

좋은 PR에는 보통 다음이 들어갑니다.

- 무엇을 바꿨는가
- 왜 바꿨는가
- 어떻게 테스트했는가
- 리뷰어가 특히 봐야 할 부분은 무엇인가
- 스크린샷이나 실행 결과가 필요한가

AI가 만든 변경사항일수록 PR 설명이 더 중요합니다.

예를 들어 PR 설명에 이렇게 적을 수 있습니다.

```markdown
## 변경 내용
- 로그인 실패 시 서버 에러 메시지를 사용자에게 표시하도록 수정
- 기존 alert 방식 대신 LoginError 컴포넌트 사용
- 관련 테스트 2개 추가

## 테스트
- npm test
- Chrome에서 잘못된 비밀번호 입력 흐름 확인

## 리뷰 포인트
- 에러 메시지 노출 범위가 보안 정책에 맞는지 확인 필요
```

이 정도만 적어도 리뷰 품질이 훨씬 좋아집니다.

### 슬라이드 15. AI로 PR 설명을 만들 때 주의할 점

AI는 PR 설명을 잘 만들어줍니다. 하지만 그대로 붙여넣기 전에 반드시 확인해야 합니다.

AI는 실제로 실행하지 않은 테스트를 실행했다고 쓸 수 있습니다. 실제로 바꾸지 않은 내용을 바꿨다고 말할 수도 있습니다.

그래서 AI에게 PR 설명을 맡길 때는 이렇게 요청하는 것이 좋습니다.

```text
현재 git diff를 기준으로 PR 설명 초안을 작성해줘.
실제로 확인되지 않은 테스트는 "미확인"으로 표시해줘.
추측하지 말고 diff에서 확인되는 내용만 써줘.
```

그리고 최종 PR 설명은 사람이 검토해야 합니다.

AI를 문서화 보조 도구로 쓰는 것은 좋습니다. 하지만 최종 책임자는 사람입니다.

## 33:00-40:00 실무 팁

### 슬라이드 16. 좋은 커밋 단위

좋은 커밋은 너무 크지도, 너무 작지도 않아야 합니다.

기준은 "나중에 이 커밋만 되돌려도 의미가 있는가?"입니다.

좋은 커밋 단위 예시는 다음과 같습니다.

- 로그인 실패 메시지 표시 버그 수정
- 사용자 입력 validation 함수 분리
- 검색 결과 empty state UI 추가
- 결제 API timeout 테스트 추가

나쁜 커밋 단위는 다음과 같습니다.

- 여러 기능, 리팩터링, 포맷팅이 한 번에 섞인 커밋
- 동작 변경과 코드 정리가 섞인 커밋
- 임시 로그와 실제 수정이 섞인 커밋

AI는 종종 "겸사겸사" 많은 것을 고칩니다. 그래서 사람이 커밋 단위를 잘 나눠야 합니다.

VSCode의 partial staging 기능을 사용하면 한 파일 안에서도 일부 변경만 커밋할 수 있습니다.

### 슬라이드 17. 커밋 메시지 패턴

팀에서 정해진 규칙이 없다면 다음 정도의 패턴만 써도 충분합니다.

```text
type: summary
```

예시는 다음과 같습니다.

```text
fix: handle expired session redirect
feat: add empty state for search results
refactor: extract user validation helper
test: cover payment timeout case
docs: update setup guide
```

한국어로 쓴다면 이렇게도 가능합니다.

```text
fix: 만료 세션 리다이렉트 오류 수정
feat: 검색 결과 없음 화면 추가
refactor: 사용자 검증 로직 분리
test: 결제 타임아웃 테스트 추가
docs: 개발 환경 설정 문서 업데이트
```

여기서 중요한 것은 `type`보다 summary입니다. summary는 구체적이어야 합니다.

`fix: bug`보다 `fix: prevent duplicate payment submission`이 훨씬 좋습니다.

### 슬라이드 18. 충돌은 실패가 아니라 대화다

Git을 쓰다 보면 conflict, 즉 충돌을 만나게 됩니다.

충돌은 누군가 잘못했다는 뜻이 아닙니다. 같은 파일의 같은 부분을 서로 다르게 바꿨기 때문에 Git이 자동으로 판단하지 못하는 상황입니다.

VSCode는 충돌 해결 UI를 잘 제공합니다.

충돌이 생기면 보통 이런 선택지가 보입니다.

- Accept Current Change
- Accept Incoming Change
- Accept Both Changes
- Compare Changes

여기서 current는 내가 현재 작업 중인 브랜치의 변경이고, incoming은 병합하려는 쪽의 변경입니다.

중요한 것은 버튼을 기계적으로 누르지 않는 것입니다.

충돌 해결은 "둘 중 하나 고르기"가 아니라 "최종적으로 맞는 코드를 만드는 일"입니다.

AI에게 충돌 해결을 맡길 수도 있지만, 반드시 사람이 최종 diff를 봐야 합니다. 충돌 해결 과정에서 비즈니스 로직이 사라지는 일이 자주 생기기 때문입니다.

### 슬라이드 19. .gitignore와 비밀정보

Git에 올리면 안 되는 파일도 있습니다.

대표적으로 다음과 같습니다.

- `.env` 같은 환경변수 파일
- API key, token, password가 들어 있는 파일
- 빌드 결과물
- 로컬 캐시
- 개인 IDE 설정 중 팀과 공유할 필요가 없는 것

이런 파일은 `.gitignore`에 추가합니다.

AI 시대에는 이 부분이 더 중요합니다. AI가 예제 코드를 만들면서 임시 키나 설정 파일을 생성할 수 있고, 사용자가 모르고 커밋할 수도 있습니다.

커밋 전에 Source Control 패널에서 새 파일 목록을 꼭 확인해야 합니다.

특히 `U`, 즉 untracked 파일은 조심해서 봐야 합니다. 새로 생긴 파일 중에 올리면 안 되는 파일이 섞여 있을 수 있습니다.

### 슬라이드 20. 히스토리는 팀의 기억이다

Git 히스토리는 단순 로그가 아닙니다. 팀의 기억입니다.

좋은 히스토리는 다음 질문에 답해줍니다.

- 이 기능은 언제 들어왔는가?
- 왜 이런 예외 처리가 있는가?
- 어떤 버그를 고치기 위해 이 코드가 생겼는가?
- 문제가 생겼을 때 어느 커밋부터 확인해야 하는가?

AI가 코드를 많이 만들어도, 히스토리를 정리하는 일은 여전히 사람의 몫입니다.

그래서 우리는 커밋을 잘게 나누고, 메시지를 분명히 쓰고, PR 설명을 남기고, 리뷰를 통해 맥락을 보존해야 합니다.

## 40:00-45:00 마무리

### 슬라이드 21. AI 시대 Git 체크리스트

마무리로 오늘 내용을 체크리스트로 정리해보겠습니다.

AI에게 작업을 맡기기 전:

- 현재 Git 상태가 깨끗한가?
- 필요한 변경사항은 미리 커밋했는가?
- 새 브랜치에서 작업하고 있는가?
- AI에게 요청할 작업 범위가 충분히 작은가?

AI가 작업한 후:

- 변경된 파일 목록을 확인했는가?
- diff를 직접 읽었는가?
- 의도하지 않은 삭제나 설정 변경이 없는가?
- 테스트나 실행 확인을 했는가?
- 의미 있는 단위로 stage했는가?
- 커밋 메시지가 구체적인가?

협업 전:

- PR 설명에 변경 이유와 테스트 내용을 적었는가?
- 리뷰어가 봐야 할 부분을 표시했는가?
- AI가 작성한 설명에 과장이나 추측이 없는가?

### 슬라이드 22. 오늘의 결론

오늘의 결론은 간단합니다.

**AI가 코드를 더 빨리 만들수록, Git은 더 중요해집니다.**

AI는 작성 속도를 높여줍니다. 하지만 Git은 그 속도를 안전하게 다룰 수 있게 해줍니다.

VSCode는 그 과정을 눈으로 확인하기 쉽게 도와줍니다. Source Control 패널에서 변경사항을 보고, diff를 읽고, stage를 나누고, 커밋하고, 브랜치와 PR로 협업할 수 있습니다.

앞으로 AI 도구는 더 좋아질 것입니다. 하지만 좋은 개발자의 기준은 단순히 AI에게 코드를 많이 뽑아내는 사람이 아닐 것입니다.

좋은 개발자는 변경을 이해하고, 검토하고, 설명하고, 필요하면 되돌릴 수 있는 사람입니다.

Git은 그 능력을 가장 현실적으로 만들어주는 도구입니다.

오늘 발표를 듣고 딱 하나만 가져가신다면 이것이면 좋겠습니다.

**AI가 코드를 바꾼 뒤에는 반드시 Git diff를 읽자.**

이 습관 하나가 많은 사고를 막아줍니다.

감사합니다.

## 부록 A. 발표 중 데모 흐름 예시

시간이 허용된다면 5분 정도 짧은 데모를 넣을 수 있습니다.

### 데모 1. VSCode에서 변경사항 확인

1. 파일 하나를 수정한다.
2. Source Control 패널을 연다.
3. 변경된 파일을 클릭해 diff를 보여준다.
4. 일부 변경만 stage한다.
5. 커밋 메시지를 작성하고 커밋한다.

설명 멘트:

"여기서 핵심은 커밋 버튼을 누르는 것이 아니라, 커밋 전에 diff를 읽는 것입니다. 내가 의도한 변경인지 확인하고, 이번 커밋에 들어갈 것만 stage합니다."

### 데모 2. 브랜치 만들기

1. VSCode 왼쪽 아래 브랜치 이름 클릭
2. Create new branch 선택
3. `feature/demo-change` 입력
4. 변경 후 커밋

설명 멘트:

"브랜치는 거창한 기능이 아니라 작업 공간을 분리하는 방법입니다. AI에게 실험적인 수정을 맡길 때는 main에서 바로 하지 말고 브랜치를 하나 만들어두면 마음이 훨씬 편합니다."

### 데모 3. AI 결과 검토하기

1. AI가 수정한 것처럼 여러 줄 변경을 만든다.
2. 파일 목록을 먼저 확인한다.
3. diff에서 삭제된 부분을 중심으로 본다.
4. 관련 없는 변경은 discard하거나 별도 커밋으로 분리한다.

설명 멘트:

"AI 결과물은 완성품이라기보다 제안된 변경사항입니다. 우리는 이 제안을 Git diff로 읽고, 필요한 것만 받아들이는 편집자가 되어야 합니다."

## 부록 B. 발표자가 기억하면 좋은 한 줄 요약

- Git은 백업 도구가 아니라 변경을 설명하는 도구다.
- AI 시대에는 코드 작성보다 변경 검토가 더 중요해진다.
- VSCode Source Control 패널은 매일 보는 대시보드처럼 써야 한다.
- stage는 이번 커밋에 담을 변경을 고르는 과정이다.
- 커밋 메시지는 미래의 나에게 보내는 메모다.
- 브랜치는 실험실이고, main은 안정된 공간이다.
- PR은 코드 리뷰 요청이면서 변경 설명 문서다.
- AI가 만든 PR 설명은 반드시 사실 확인해야 한다.
- 충돌 해결은 버튼 선택이 아니라 최종 코드를 다시 구성하는 일이다.
- 커밋 전에는 항상 새 파일과 비밀정보를 확인한다.
- 결론: AI가 코드를 바꾼 뒤에는 반드시 Git diff를 읽자.
